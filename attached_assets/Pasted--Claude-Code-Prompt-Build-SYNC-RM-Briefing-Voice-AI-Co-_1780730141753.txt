## PROJECT BRIEF

Build a production-ready hackathon project called **"SYNC"** — a Voice AI co-pilot for bank/NBFC Relationship Managers. An RM calls a phone number (or uses a web widget) before a client meeting, says the client's name, and receives a crisp 45-second voice briefing covering portfolio health, risk flags, open complaints, and a personalized cross-sell recommendation.

**This is for the GrowthX Voice AI Buildathon (powered by Ringg AI).** Ringg AI is the mandatory voice platform.

#### 1.5 `backend/routers/calls.py`

```
POST /api/v1/calls/sync-now
     Body: { "client_id": "...", "rm_phone": "+919...", "rm_name": "..." }
     
     Flow:
     1. Fetch full client profile from CRM adapter
     2. Generate briefing text using briefing_engine (GPT-4o)
     3. Call Ringg API: POST /calling/outbound/individual
        - agent_id: RINGG_AGENT_ID
        - from_number_id: RINGG_FROM_NUMBER_ID
        - mobile_number: rm_phone (the RM's number — SYNC calls the RM)
        - name: rm_name
        - custom_args_values: {
            "callee_name": rm_name,
            "client_name": client.name,
            "client_age": client.age,
            "client_occupation": f"{client.occupation} at {client.company}",
            "portfolio_summary": <generated summary string>,
            "risk_level": risk.score,
            "risk_factors": ", ".join(risk.factors),
            "days_since_contact": last_interaction_days,
            "open_complaints": <complaint summary or "None">,
            "cross_sell_pitch": <primary cross-sell pitch_angle>,
            "cross_sell_product": <primary product name>,
            "secondary_pitch": <secondary cross-sell if exists>,
            "hinglish_closer": <a natural hinglish sign-off>
          }
        - callback_url: f"{BACKEND_URL}/api/v1/webhooks/ringg"
     4. Store call_id in memory
     5. Return { "call_id": "...", "status": "initiated" }
```

#### 1.6 `backend/routers/webhooks.py`

#### 1.8 `backend/adapters/base.py`

```python
from abc import ABC, abstractmethod

class CRMAdapter(ABC):
    """Abstract base class for CRM integrations."""
    
    @abstractmethod
    async def search_client(self, name: str) -> list[ClientProfile]:
        """Fuzzy search clients by name."""
        pass
    
    @abstractmethod
    async def get_client(self, client_id: str) -> ClientFullProfile:
        """Get full client profile with all data."""
        pass
    
    @abstractmethod
    async def get_portfolio(self, client_id: str) -> list[LoanProduct]:
        pass
    
    @abstractmethod
    async def get_risk(self, client_id: str) -> RiskAssessment:
        pass
    
    @abstractmethod
    async def get_cross_sell(self, client_id: str) -> list[CrossSellOpportunity]:
        pass
    
    @abstractmethod
    async def get_interactions(self, client_id: str) -> tuple[list[Interaction], list[Complaint]]:
        pass
    
    @abstractmethod
    async def log_briefing(self, briefing: BriefingLog) -> None:
        """Log a completed briefing back to CRM."""
        pass
```

#### 2.2 `ringg/system-prompt.md`

Full system prompt for the Ringg agent (this goes into `introduction_and_objective` + `response_guidelines` + `task` fields):

```markdown
# You are SYNC

You are a voice AI co-pilot for Relationship Managers at an Indian bank/NBFC.
An RM calls you before a client meeting. Your job: deliver a crisp, warm,
45-second briefing so the RM walks in fully prepared.

## Your personality
- Sound like a sharp, friendly colleague — not a bot, not a call center
- Confident but not robotic. Use natural fillers ("So here's the thing about Rahul...")
- Speak in English by default, but naturally code-switch to Hinglish
  when it fits ("EMI time pe aa raha hai, no issues there")
- NEVER sound like you're reading a report — TELL the story of a person

## For OUTBOUND calls (dashboard triggered):
- You already have client data in your custom variables
- Start with: "Hey @{{callee_name}}! Got your sync ready for @{{client_name}}. Here goes..."
- Deliver the briefing using the custom variable data

## For INBOUND calls (RM called):
- Start with: "Hey! This is SYNC. Which client are you meeting today?"
- When RM says a name, look it up in your knowledge base
- If found, deliver the briefing
- If not found, say: "I'm not finding anyone by that name — could you spell it out or give me a last name?"

## Briefing structure (ALWAYS this order):
1. Identity anchor (5 sec): Name, age, job, one human detail
2. Portfolio snapshot (10 sec): Products, amounts, EMI status — narrated, NOT listed
3. Risk flag (5 sec): Red/amber signals said plainly. If HIGH, lead the whole briefing with this
4. Relationship gap (5 sec): Days since last contact + open complaints
   - If there's an open complaint, ALWAYS mention it (worst thing = RM walks in unaware)
5. The play (15 sec): What to pitch and WHY — tied to client's life context, not just eligibility

## Rules
- NEVER exceed 60 seconds for the full briefing
- If risk is HIGH, restructure: lead with risk, then portfolio, then the play
- Handle interruptions gracefully — answer the question, offer to continue
- If you don't know something: "I don't have visibility on that, might want to check the system"
- End with: "That's the quick sync. Anything specific, or are you good to walk in?"
- If they're good: warm Hinglish sign-off + "I'll log this touchpoint"
```

#### 3.2 Components

**MetricCards.jsx**
- 4 stat cards: Syncs Today, Avg Time Saved, Cross-sells Surfaced, Complaints Flagged
- Initial values hardcoded but update in real-time via WebSocket
- Subtle animation when a number increments

**ClientSelector.jsx**
- Dropdown with search (type-ahead) that queries `GET /api/v1/clients`
- Shows client name, company, risk badge (colored dot)
- On select, show a mini preview card: name, products, risk level

**SyncNowButton.jsx**
- Takes selected client_id and rm_phone
- POSTs to `/api/v1/calls/sync-now`
- Shows spinner → "Calling..." → "Connected" → "Sync delivered ✓"
- Transitions based on WebSocket events

**LiveFeed.jsx**
- Reverse chronological list of BriefingLog events
- Each card: timestamp, RM name, client name, risk badge, complaint badge, pitch summary, duration, status
- New events slide in from top with subtle animation
- Pre-seed with 3-4 fake historical events so the dashboard doesn't look empty on demo start

**BeforeAfter.jsx**
- Simple two-column comparison
- Static content, no interactivity needed

**LatencyBadge.jsx**
- Shows "Pipeline: 340ms" or similar
- Updates per call from webhook data
- Green if <500ms, yellow if <800ms, red if >800ms

## IMPLEMENTATION PRIORITIES

Build in this order:

1. **Backend core** — models, database, client endpoints, health check
2. **CRM adapter interface** — base class + mock implementation
3. **Ringg service** — API client for agent creation, call initiation, webhooks

## CODE QUALITY REQUIREMENTS

- Type hints on all Python functions
- Pydantic models for all request/response schemas
- Proper error handling with meaningful HTTP status codes
- Logging with structlog
- CORS configured for dashboard origin
- Environment variables for all secrets (never hardcode)
- Docstrings on all public functions
- Clean separation of concerns (routers → services → adapters)

## KEY TECHNICAL DECISIONS

- Use `httpx.AsyncClient` (not requests) for all external API calls
- Use `rapidfuzz` for fuzzy name matching (not difflib — it's faster)
- Use `websockets` or FastAPI's built-in WebSocket for dashboard updates
- Dashboard uses Vite + React + Tailwind (no Next.js, keep it simple)
- All money amounts stored as integers (paise), displayed as rupees
- Dates in ISO 8601, displayed in IST
